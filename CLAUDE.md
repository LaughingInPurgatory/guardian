# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Guardian is a modern, endless remake of the arcade game *Defender*, built as an Electron app. It has **zero runtime dependencies beyond Electron itself**: all graphics are drawn on `<canvas>`, all sound effects and the chiptune soundtrack are synthesized live with the Web Audio API, and high scores persist via `localStorage`. There are no image/audio asset files and no game engine/audio library — this is deliberate; keep it that way rather than reaching for a dependency.

## Commands

```
npm install
npm start          # run the Electron app
npm test           # run the logic test suite (node --test)
npm run dist        # build an installer for the current platform via electron-builder
```

- To run a single test file directly: `node --test test/logic.test.js`
- There is no lint/typecheck script configured.
- `node --check src/game.js` (or any file) is a quick syntax-only sanity check without running the app.

### Releasing

Pushing a tag matching `v*` triggers `.github/workflows/release.yml`, which builds mac (`.dmg`, arm64+x64), Linux (`.AppImage`, arm64+x64), and Windows (`.exe`, x64+arm64) in a matrix and merges them into a single GitHub Release. **Do not let matrix jobs publish directly** — GitHub does not enforce tag uniqueness on draft releases, so concurrent `electron-builder --publish` calls from parallel OS jobs will create duplicate drafts instead of sharing one (this happened once; the fix was to have each job upload build artifacts via `actions/upload-artifact` and have one downstream job merge them with `softprops/action-gh-release`). Keep that structure.

The macOS build is ad-hoc signed (`scripts/afterSign.js`, since there's no paid Apple Developer ID) with `hardenedRuntime`/`gatekeeperAssess` disabled. It is not notarized, so Gatekeeper will still quarantine it on download; the README documents the `xattr -cr` workaround.

The app icon (`build/icon.png`) is generated by `scripts/generate-icon.js`, which renders the ship on a canvas inside an offscreen Electron `BrowserWindow` and captures it via `webContents.capturePage()` — this avoids needing an image-editing tool or a new dependency. Re-run it with `npx electron scripts/generate-icon.js` if the icon design changes; the output PNG is committed to the repo, not regenerated at build time.

## Architecture

**Process split**: `main.js` is the Electron main process (creates the single `BrowserWindow`, handles `quit-to-desktop` over IPC). `preload.js` exposes exactly one bridged function (`window.guardian.quitToDesktop()`) via `contextBridge`; `contextIsolation` is on and `nodeIntegration` is off, so the renderer has no Node access beyond that one call.

**Renderer module loading**: There is no bundler. `index.html` loads `src/logic.js`, `src/audio.js`, `src/highscores.js`, `src/puns.js`, `src/game.js` as plain `<script>` tags, in that order (dependency order matters). Each file wraps its body in an IIFE and attaches its exports to `window.Guardian*` (e.g. `window.GuardianLogic`, `window.GuardianAudio`) — **this IIFE wrapping is required**, not stylistic: classic `<script>` tags share one global lexical scope, so un-wrapped top-level `const`/`class` declarations collide if the page context is ever re-entered (this broke once). `logic.js` additionally does `module.exports` so the same file works unmodified under `node --test`.

**Module responsibilities**:
- `src/logic.js` — pure, dependency-free math (`wrap`, `wrapDelta` for the wraparound world, `circleCollision`, `computeDifficulty`). The only module covered by the test suite (`test/logic.test.js`); keep new pure/testable logic here rather than buried in `game.js`.
- `src/audio.js` — `AudioEngine` class. Every SFX is synthesized via short-lived `OscillatorNode`/noise-buffer graphs with hand-built envelopes (see `_tone`/`_noiseBurst`). Music is a procedural step-sequencer using the standard Web-Audio lookahead-scheduler pattern (`_musicScheduler`, tick every 25ms, schedule ~120ms ahead) over a small set of scale/pattern variants selected by wave number. `init()` must be called from a user-gesture handler (browser autoplay policy) and also auto-starts the music loop.
- `src/highscores.js` / `src/puns.js` — trivial `localStorage` wrapper and a static pun list.
- `src/game.js` — the entire game engine in one `Game` class (~1150 lines). This is the file you'll spend the most time in.

**`Game` class structure** (all in `src/game.js`): a single `requestAnimationFrame` loop (`_loop` → `_update`/`_render`) driving a state machine (`title` / `playing` / `paused` / `gameover`) that toggles CSS visibility on overlay `<div>`s in `index.html` rather than drawing UI on canvas. Method naming convention: `_update*`/`_draw*` pairs per subsystem (enemies, bullets, humanoids, powerups, particles, shockwaves, player bombs, turrets, minimap, HUD).

Key mechanics worth knowing before touching them:
- **World wraparound**: the world is a fixed-width ring (`WORLD_W`); almost everything horizontal must go through `wrap()`/`wrapDelta()` rather than raw subtraction, or wraparound collisions/rendering will be wrong near the seam. `_relX()` converts a world x to a camera-relative screen x; `_wrappedHit()` is the wraparound-aware version of `circleCollision`.
- **Terrain**: generated once per game (`generateTerrain`) as an elevation array sampled with `groundHeightAt()`; enemy/humanoid spawn Y must be clamped against this per-x height (see `GROUND_CLEARANCE`) rather than a flat baseline, or things spawn underground.
- **Player physics**: `PLAYER_DRAG` is `0` deliberately — this is authentic Defender-style inertia (no passive deceleration; only active reverse-thrust brakes you). Don't add drag back without discussing it.
- **Death/respawn**: `_killPlayer(cause)` distinguishes `'ground'` vs `'combat'` deaths — ground-collision deaths respawn raised to mid-screen (crashing at the same spot would just crash again), combat deaths respawn exactly where they died. Respawn is a two-phase delay (explosion → implosion-particle reform) driven by `player.respawnTimer`/`player.hidden`, not an instant relocation.
- **Difficulty**: uncapped/endless by design (`computeDifficulty` in `logic.js` — speed, spawn rate, and enemy cap all grow linearly forever). Don't add a ceiling; the game is meant to eventually become unsurvivable.
- **Input**: keyboard (WASD/arrows), gamepad, and mouse-cursor steering are all read and combined in `_readCombinedInput()` every frame. Mouse steering is gated behind `this.mouseControlEnabled` (toggled with `M`, off by default) to avoid incidental cursor drift fighting keyboard/gamepad input — keep that gate if you touch input handling. Game keybindings are ignored while focus is inside an `<input>` (the high-score initials field).
- **Particles**: a single generic `this.particles` array (drawn with additive/`lighter` blending) is reused for exhaust, explosions, and the implosion-reform effect; `this.shockwaves` is a separate expanding-ring effect layered on top for bigger hits (player death, smart bomb, drop bomb).
